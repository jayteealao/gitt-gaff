# File Naming Conventions

All workspace files follow a consistent naming pattern for organization and discoverability.

## Standard Format

```
{YYYY-MM-DD}-{kebab-case-description}.md
```

**Components:**
- **YYYY-MM-DD** - Creation date (ISO 8601)
- **kebab-case-description** - Lowercase words separated by hyphens
- **.md** - Markdown extension

## Description Guidelines

### Do
- Use descriptive, searchable terms
- Keep it concise (3-6 words ideal)
- Use domain-specific terminology
- Make the purpose clear from the name

### Don't
- Use vague names like `notes.md` or `todo.md`
- Include status in filename (use frontmatter)
- Use abbreviations that aren't widely known
- Make names too long (>80 chars total path)

**Good names:**
```
2025-01-07-user-authentication-migration.md
2025-01-07-postgres-to-mysql-migration.md
2025-01-07-react-query-adoption.md
```

**Bad names:**
```
2025-01-07-plan.md                    # Too vague
2025-01-07-wip-auth-stuff.md          # Status in name, informal
2025-01-07-usr-auth-mig-impl-v2.md    # Excessive abbreviations
```

## Special Files

### INDEX.md

Each category has exactly one `INDEX.md` file:
```
plans/INDEX.md
architecture/INDEX.md
examples/INDEX.md
research/INDEX.md
analysis/INDEX.md
```

INDEX.md files:
- Do NOT follow date prefix convention
- Are auto-generated by the update-index workflow
- Should not be manually edited (will be overwritten)

---

## Date Usage

### Why Date Prefix?

1. **Natural sorting** - Files sort chronologically by default
2. **Context** - Know when a document was created at a glance
3. **Uniqueness** - Prevents name collisions
4. **History** - Track document evolution over time

### Date Rules

- Use creation date, not modification date
- Use local date (not UTC unless team convention)
- If creating multiple files on same day with same topic, add suffix:
  ```
  2025-01-07-auth-research.md
  2025-01-07-auth-research-oauth.md
  2025-01-07-auth-research-saml.md
  ```

## Kebab Case Rules

### Conversion

| Original | Kebab Case |
|----------|------------|
| User Authentication | user-authentication |
| API v2 | api-v2 |
| N+1 Query | n-plus-one-query |
| PostgreSQL to MySQL | postgresql-to-mysql |
| OAuth 2.0 | oauth-2-0 |

### Rules

1. **Lowercase everything** - `User` becomes `user`
2. **Replace spaces with hyphens** - `user auth` becomes `user-auth`
3. **Remove special characters** - Except hyphens
4. **Numbers stay** - `v2` stays `v2`
5. **Spell out symbols** - `N+1` becomes `n-plus-one`

## Renaming Files

When renaming workspace files:

1. **Keep original date** - Don't change the creation date
2. **Update INDEX.md** - Run update-index workflow
3. **Update references** - Fix any links pointing to old name
4. **Update frontmatter** - Ensure title matches new description

**Example:**
```
# Renaming for clarity
OLD: 2025-01-07-auth.md
NEW: 2025-01-07-oauth-provider-integration.md

# Steps:
1. Rename file
2. Update title in frontmatter
3. Search for links to old filename
4. Update INDEX.md
```

## Path Length Limits

Keep total path length reasonable:

```
.claude/architecture/2025-01-07-microservices-communication-patterns-for-distributed-systems.md
        ^----------^ ^--------^ ^---------------------------------------------------------------^
        11 chars     10 chars   67 chars = 88 chars total

# This is getting long - consider shortening:
.claude/architecture/2025-01-07-microservices-communication.md
# 52 chars total - much better
```

**Guidelines:**
- Description: 3-6 words (20-50 chars)
- Total filename: <80 chars
- Full path from repo root: <150 chars
